# Tasks v1 Spec

**Date:** 2026-02-14
**Status:** Draft

## Overview

Tasks represent units of work on a project. They provide structure above sessions, giving a place to track background/async agent work and surface what needs human attention. Sessions can belong to a task or exist as standalone scratch sessions.

## Concepts

### Task
A named unit of work on a project. Owns zero or more sessions. Each task gets its own git branch, creating isolation from the main codebase.

### Scratch Session
A session with no task association. Operates on whatever branch is currently checked out in the project working tree. Interactive and ephemeral — same as sessions work today.

### Task Session
A session belonging to a task. Operates on the task's branch. The task's title and description are injected into the session's system prompt so the agent has context about what it's contributing to.

## Task Lifecycle

No formal state machine for v1. Tasks exist and can be deleted when done. Status tracking (active, awaiting review, done) is a future addition.

## Git Integration

- When a task is created, a branch is created from the project's `base_branch` — regardless of what branch is currently checked out (e.g., `git branch <name> <base_branch>`). The branch name is generated by an LLM call (e.g., Haiku) from the task title, producing a clean slug like `task/refactor-auth-middleware`. If the branch name collides with an existing branch, creation fails.
- All sessions under a task operate on that branch. The task branch is checked out when a task session starts; no automatic stashing or cleaning of the working tree is performed.
- The Changes tab can show a diff of any task's branch against the project's base branch (`base_branch..task_branch`), without needing to check out the branch.
- Deleting a task does **not** delete its git branch.

## Data Model

### New `tasks` table

| Column       | Type    | Notes                                      |
|--------------|---------|--------------------------------------------|
| id           | INTEGER | PK, autoincrement                          |
| project_id   | INTEGER | FK → projects                              |
| title        | TEXT    | Required                                   |
| description  | TEXT    | Nullable, optional longer context          |
| branch_name  | TEXT    | Git branch name                            |
| created_at   | TEXT    | Default `datetime('now')`                  |
| updated_at   | TEXT    | Default `datetime('now')`                  |

### Sessions table changes

| Column  | Type    | Notes                              |
|---------|---------|------------------------------------|
| task_id | INTEGER | Nullable FK → tasks. Null = scratch. |

## System Prompt Injection for Task Sessions

When a session belongs to a task, prepend to the system prompt:

```
## Task
Title: {task.title}
Description: {task.description}

You are working on this task.
```

## UI Changes

### Sidebar (primary view: tasks)

The sidebar replaces the current session list with a task-oriented view:

- **Tasks** section — listed by most recently updated.
- **Scratch sessions** section (bottom) — standalone sessions, same as today's session list.

Each task item shows:
- Title
- Number of sessions
- Last updated time

Clicking a task expands it to show its sessions. Clicking a session opens it in the chat panel.

### Task Actions

- **New Task** button — creates a task with a title (and optional description), creates the git branch.
- **New Session** on a task — creates a session under that task.
- **Delete Task** — removes the task and its sessions.

### Changes Tab

- When viewing a task session, the Changes tab diffs the task's branch against the project's base branch.
- When viewing a scratch session, behavior unchanged (diffs working tree).

## API Endpoints

### Tasks

- `GET    /api/projects/:projectId/tasks` — list tasks
- `POST   /api/projects/:projectId/tasks` — create task `{ title, description? }`, creates git branch
- `GET    /api/projects/:projectId/tasks/:taskId` — get task with its sessions
- `PATCH  /api/projects/:projectId/tasks/:taskId` — update title, description
- `DELETE /api/projects/:projectId/tasks/:taskId` — delete task (and its sessions)

### Task Sessions

- `POST   /api/projects/:projectId/tasks/:taskId/sessions` — create session under task
- `GET    /api/projects/:projectId/tasks/:taskId/sessions` — list sessions for task

### Scratch Sessions (unchanged)

- `POST   /api/projects/:projectId/sessions` — create scratch session (no task_id)
- `GET    /api/projects/:projectId/sessions` — list scratch sessions only

## Out of Scope for v1

- Task status lifecycle (active, awaiting review, done) and associated tooling.
- Concurrent task isolation (worktrees, sandboxing) — handled operationally for now.
- Automated PR creation from tasks.
- Agent-initiated task creation.
- Cross-project task view (sidebar is still project-scoped).
- Moving sessions between tasks.
- Primary session concept within a task.
